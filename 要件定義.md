# 要件定義書（RSS技術記事スコアリング & 再配信システム）

## 1. 背景・目的
最新かつ価値の高い技術記事を複数サイトから自動収集し、LLM（ChatGPT API）で記事ごとに評価・ランキングした上で、上位記事のみを再配信する RSS を生成・ホスティングする。情報探索コストを削減し、重要トピックを定時（朝 7:00 / 夕方 17:00）に取得できる仕組みを提供する。

## 2. スコープ
- 対象: 公開 RSS フィードを持つ技術系サイト（初期 3 種）
- 出力: ランキング済み上位 30 件の RSS フィード（RSS 2.0）
- 更新頻度: 1 日 2 回（07:00 / 17:00 JST）
- 評価: LLM による 3 観点（新規性 / 興味深さ / 専門性, 各 10 点）＋合計点

## 3. 用語定義
| 用語 | 定義 |
|------|------|
| フィード元 | 収集対象となる外部 RSS URL |
| アイテム | RSS の individual entry (item) |
| スコアリング | LLM により 3 観点へ点数付与する処理 |
| 正規化 | 異なる RSS のフィールドを内部統一形式へ変換 |
| 出力フィード | 当システムが生成するランキング済み RSS |

## 4. 利用想定シナリオ
1. 毎朝 7 時 / 夕方 17 時に新しいフィードが生成される。
2. 利用者は任意の RSS リーダーへ購読登録し、上位 30 件のみを素早く確認。
3. 既読管理などは RSS リーダー側に委譲。

## 5. 機能要件（機能一覧）
### 5.1 収集
- 対象フィード初期値:
  - https://zenn.dev/feed
  - https://codezine.jp/rss/new/20/index.xml
  - https://qiita.com/popular-items/feed
- HTTP 取得（タイムアウト / リトライ / User-Agent 指定）
- フィード取得失敗時は部分的成功を許容し、失敗ログを残す

### 5.2 正規化 / 前処理
- RSS/Atom 形式を内部統一モデルへマッピング
  - タイトル, リンク, 公開日時 (pubDate), サマリ/本文（可能なら description or content:encoded）
- 公開日時の標準化 (UTC ISO8601)
- 重複判定: link または (title + pubDate) ハッシュで排重

### 5.3 スコアリング
- LLM API を利用し、1 アイテムごとに以下を 0〜10 点整数で出力: 新規性, 興味深さ, 専門性
- 合計点 = 3 観点の単純合計 (最大 30)
- プロンプト内で望ましい JSON フォーマットを強制 (例: {"novelty": int, "interest": int, "expertise": int, "reason": "..."})
- 出力バリデーション: 値が欠損/範囲外/非整数の場合は再試行（最大 2 回）
- レート制御: 並列数上限設定 + 過剰課金防止カウンタ

### 5.4 ランキング
- 合計点降順、同点時は (新規性 > 専門性 > 興味深さ) の順、さらに同一なら新しい pubDate 先。

### 5.5 出力 RSS 生成
- 上位 30 件を RSS 2.0
- channel 要素: title, link, description, language (ja), lastBuildDate
- item 要素へスコアを付加（例: <category>Score: 27 (N=9/I=9/E=9)</category> もしくは <dc:subject> 等拡張）
- キャッシュ制御: ETag / Last-Modified ヘッダ返却（ホスティング方式に依存）

### 5.6 スケジューリング
- 07:00 / 17:00 JST 自動実行
- 手動トリガ（再生成）オプション（将来）

### 5.7 監視 / ログ
- 収集件数, 失敗件数, LLM 呼出件数, コスト見積をメトリクス化
- エラー通知（将来: Slack / Email）

### 5.8 設定管理
- フィード URL リストを設定ファイル (YAML/JSON) または環境変数にて管理し拡張可能

## 6. 非機能要件
| 区分 | 要件 |
|------|------|
| パフォーマンス | 3 フィード + 合計 100〜300 item 想定を 2〜5 分以内処理 |
| 可用性 | 実行スロット毎に 90% 以上成功（失敗時は次回で復旧） |
| 拡張性 | フィード数 10, item 1000 程度までコード改修なし（並列度設定変更のみ） |
| セキュリティ | API Key を環境変数 / Secret Manager に保存。リポジトリへ平文禁止。 |
| 運用性 | ログ参照 1 分以内、再実行コマンド簡易化。 |
| コスト | 月数ドル以内（無料枠活用）。 |
| 国際化 | 初期 日本語メイン、将来英語説明付与オプション。 |
| プライバシー | 外部公開記事のみ扱い。個人情報なし。 |

## 7. アーキテクチャ概要
1. スケジューラ（CRON）起動
2. RSS フェッチ（並列）→ 正規化 → 重複排除
3. LLM スコアリング（バッチ / 並列制御）
4. ソート → 上位 30 抽出
5. RSS XML 生成
6. ストレージへ配置（静的配信）
7. 成功メトリクス記録

## 9. LLM スコアリング詳細
### 9.1 プロンプト設計（例）
System: あなたは技術記事を評価する専門アナリストです。以下の JSON 形式のみで出力してください。
User: 以下の記事について「技術の新規性」「興味深さ」「専門性」を 0〜10 の整数で評価し、最後に理由を 100 文字以内で日本語で述べてください。\nタイトル: {title}\n概要: {summary}\n本文抜粋: {excerpt}\n出力 JSON スキーマ: {"novelty":int,"interest":int,"expertise":int,"reason":"..."}

### 9.2 バリデーション
- JSON パース失敗 → 再試行
- 3 観点が整数 0..10 でない → 再試行
- 再試行上限 2 超過 → デフォルト中央値 (5) を代入 & reason="fallback"

### 9.3 コスト試算（例）
- 1 回 200 tokens 入出力, 1 実行 200 item → 40,000 tokens / 実行
- 1 日 2 回 → 80,000 tokens / 日 → 月 ~2.4M tokens（モデル単価により数ドル〜十数ドル）
- 最適化案: 低価格モデル + タイトル/概要中心, まとめバッチ（複数 item 同時）

## 10. データモデル（内部）
Item:
```
{
  id: string,            // ハッシュ(link|title|pubDate)
  source: string,        // フィード識別子
  title: string,
  url: string,
  published_at: datetime (UTC),
  summary: string,
  content_excerpt: string,
  scores: { novelty:int, interest:int, expertise:int, total:int, reason:string }
}
```

## 11. ランキングロジック
Pseudo:
```
sort(items, key=(total, novelty, expertise, interest, published_at), desc=True)
```

## 12. 出力 RSS 仕様
- RSS 2.0 準拠
- channel.title: "Tech Curated Top 30"
- item.title: 元記事タイトル
- item.link: 元リンク
- item.pubDate: RFC1123 (UTC もしくは JST 表記ポリシー統一)
- item.description: 元 summary + スコア表 + 元抜粋 + 理由
- item.guid: 内部 id (permalink=false)
- item.categories: "Score:{total}", "Novelty:{n}", 等（RSSリーダー表示状況に応じ微調整）

## 13. ホスティング選択肢比較
| 選択肢 | 長所 | 短所 | 初期判断 |
|--------|------|------|----------|
| GitHub Actions + Pages | 無料 / Secrets 管理容易 / 静的配信高速 | 実行時間/頻度制限, 並列最適化困難 | 有力(低コスト) |
| Cloudflare Workers + KV/R2 | グローバル低レイテンシ / Cron / キャッシュ | KV/R2 設計追加 | 将来拡張候補 |
| AWS Lambda + EventBridge + S3 | 柔軟 / 将来拡張容易 | 管理・課金やや複雑 | 規模拡大時 |
| Vercel Cron + Edge | 簡易 / デプロイ高速 | Free 枠実行回数制限 | 代替 |

初期案: GitHub Actions (cron) で生成→ `docs/rss.xml` へ出力→ GitHub Pages で配信。

## 14. スケジューリング仕様
- GitHub Actions cron: `0 22,8 * * *` (UTC → JST 07:00/17:00) など。
- タイムゾーン差異に注意。生成完了後コミット/Push。

## 15. ログ / モニタリング
- 標準出力: 実行開始/終了, フィード別件数, エラー要約, LLM 呼出数, 再試行数
- 将来: JSON Lines ログ → GCS/S3 or Cloudflare Logs

## 16. エラーハンドリング方針
| ケース | 対応 |
|--------|------|
| フィード 404/タイムアウト | 再試行 (2 回) → スキップ & 警告ログ |
| LLM API 429 | バックオフ（指数 + ジッタ）最大 5 分以内に収束 |
| LLM JSON 不正 | 再試行 2 回 → フォールバックスコア |
| 出力生成失敗 | 一時ファイル→アトミックリネーム採用で部分書込み防止 |
| コミット失敗 | 再試行（Git pull → merge → push） |

## 17. セキュリティ / 秘匿情報
- OpenAI API Key は環境変数 `OPENAI_API_KEY` で注入
- リポジトリへ平文コミット禁止（本ドキュメントにも記載しない）
- 監査: CI でシークレット検出ツール（gitleaks 等）導入検討
- ネットワーク: 外部呼び出しは対象 RSS + OpenAI エンドポイントのみ

## 19. リスクと対策
| リスク | 影響 | 対策 |
|--------|------|------|
| LLM コスト上振れ | 運用費増 | item 数上限 / バッチ方式 / 低価格モデル |
| RSS 構造変更 | パース失敗 | フィールド存在チェック + フォールバック |
| 重複記事多発 | ランキング精度低下 | ハッシュキー拡張（title 正規化 + link） |
| LLM 出力ばらつき | ランク不安定 | temperature 低設定 (0〜0.2) |
| Secrets 漏洩 | API 無断利用 | ローテーション手順整備 + 権限最小化 |
