# リファクタリング最終計画（部分成功許容・差分別紙・計測で安全最適化）

本書は、現行のRSS収集→正規化→スコアリング→ランキング→RSS生成パイプラインを、仕様との整合・保守性・可観測性・最適化容易性の観点でリファクタするための実行計画である。機能追加やUX追加は行わず、既存要件の範囲で「正しく・安く・安定して」動く状態を目標にする。

## 前提（確定している現状・方針）
- 入口は非同期バッチ実行で、パイプラインは [src/main.py](../src/main.py) がオーケストレーションする。
- フィードURLは [src/config.py](../src/config.py) の `FEED_URLS` が唯一のソースである（要件は 変更要件.md、原案は 要件定義.md）。
- 「過去N時間のみ」を `TIME_WINDOW_HOURS` でフィルタする（実装は [src/main.py](../src/main.py) の `filter_recent_articles`）。
- スコアリングは「文化と技術の交差点」向けの6指標（技術3＋文化3）で、プロンプトは [src/rules_prompt.txt](../src/rules_prompt.txt) を基準とする。スコアモデルは [src/models.py](../src/models.py) の `ScoreResult` に集約される。
- LLM呼び出しはバッチ優先＋多段フォールバック（API未設定/失敗→ヒューリスティック等）で、実装は [src/scorer.py](../src/scorer.py) に集約されている（背景は IMPLEMENTATION_REPORT.md）。
- 仕様差分（例: `TOP_N` の既定値、workflowのcron、フィード追加漏れ、RSSのpubDate扱いなど）は「修正予定/許容/保留」を分離して管理し、本文書では判断基準と進め方のみ定義する。差分の記録は [docs/spec_diffs.md](spec_diffs.md) に集約する。

## 目標
- 仕様に対して「どこが一致/不一致か」を明文化し、変更の優先度・影響・判断を追跡可能にする（差分管理の制度化）。
- スコアリング周り（バッチ/個別/フォールバック/キャッシュ）を読みやすく分解し、テスト可能な単位にする。
- 最適化（コスト・時間・安定性）を“測ってから”行えるように、メトリクスを定義して継続記録する（最適化の前提づくり）。

## 非目標（やらないこと）
- 新しい画面、ダッシュボード、通知チャネル等のUX/機能追加はしない（要件にある「将来」は扱わない）。
- 新しい評価軸の追加や、ランキングの思想変更はしない（既存の評価軸・並び替え規則を前提に、仕様整合のみ行う）。

## 成果物（リファクタ完了の定義）
- 差分ドキュメント: [docs/spec_diffs.md](spec_diffs.md) が埋まっており、各差分に「結論（合わせる/許容/保留）」と根拠がある。
- 回帰防止: 少なくともランキング・RSS生成・フォールバックについて、意図した挙動がテストで固定されている（既存テストは tests/）。
- メトリクス: 1実行ごとに主要メトリクスがログまたはファイルに残り、最適化前後の比較が可能。

## 「計画フェーズ」完了条件（実作業に入る前に必須）
このリポジトリでは「計画→実作業」を明確に分離する。実作業（コード/ワークフロー/運用の変更）に入る前に、最低限以下が満たされていること。

- [docs/spec_diffs.md](spec_diffs.md) の差分サマリが最新で、各差分に「結論（合わせる/許容/保留）」が入っている
- “保留”が残る場合は、必ず「次アクション」「期限」「判断に必要な追加情報」が書かれている
- 仕様が衝突する場合の優先順位が明示されている（本書の「仕様の優先順位」）
- 変更後の確認方法（テスト/メトリクス/手動確認）が差分ごとに紐づいている（実装は後でよい）

## 仕様の優先順位（衝突時の扱い）
仕様文書が複数存在するため、衝突時の優先順位を固定する。原則は「新しい変更要件を最優先」とする。

1. 変更要件.md（最新の変更指示）
2. 要件定義.md（ベース要件）
3. 設計.md（実装方針/想定）
4. DEPLOYMENT.md（運用手順の説明）

例外:
- セキュリティ/コンプライアンスに関わる場合は、運用上の安全策を優先する（例: Secrets/Variablesの扱い）
- RSSリーダー互換性のように外部仕様に依存する場合は、一次情報（仕様/validator）を根拠として採用する（差分に根拠リンクを残す）

## 進め方（フェーズ別）
### フェーズ0: 現状棚卸し（差分の見える化）
- 差分テンプレを作成し、初期差分を列挙する: [docs/spec_diffs.md](spec_diffs.md)
- 差分の出どころ（仕様参照元）を固定する
  - 基本要件: 要件定義.md
  - 変更要件（フィード/時間窓/嗜好）: 変更要件.md
  - 設計: 設計.md
  - 実装背景（任意）: IMPLEMENTATION_REPORT.md
- 実装側の“根拠リンク”を付ける（例）
  - 設定: [src/config.py](../src/config.py)
  - フェッチ/正規化: [src/fetcher.py](../src/fetcher.py)
  - スコア: [src/scorer.py](../src/scorer.py)
  - ランキング: [src/ranking.py](../src/ranking.py)
  - RSS生成: [src/rss_builder.py](../src/rss_builder.py)

達成度（部分成功を許容）
- A: 差分が網羅的（主要領域が埋まり、結論が付く）
- B: 差分が列挙済み（結論が未確定でも、根拠リンクと影響は記載）
- C: 重要差分のみ列挙（TOP_N、時間窓、フィード、RSS日付、評価軸の5点が最低限）

### フェーズ1: 境界整理（読みやすさ・責務の分離）
- パイプラインの責務を固定する（ドキュメント化してからコード調整）
  - [src/main.py](../src/main.py): “配線”のみ（取得→正規化→時間フィルタ→スコア→ソート→書き出し）
  - [src/fetcher.py](../src/fetcher.py): 取得・パース・正規化・重複排除（I/Oと整形）
  - [src/scorer.py](../src/scorer.py): プロンプト生成、API呼び出し、レスポンス検証、フォールバック、キャッシュ
  - [src/rss_builder.py](../src/rss_builder.py): RSSレンダリング（入力はRankedArticleのみ）
  - [src/ranking.py](../src/ranking.py): 並び替え規則の単一責務
- “設定値の意味”を [src/config.py](../src/config.py) に集約し、使われ方の整合を確認する（未使用/重複/命名揺れの除去）

達成度
- A: 各モジュールが単体で読め、循環参照がない
- B: 主要関数のI/Oが明確（docstring/型/責務が整理）
- C: 入口(main)から追える（追跡性だけ担保）

### フェーズ2: 仕様整合（差分ごとに判断して潰す）
- [docs/spec_diffs.md](spec_diffs.md) の各差分について、以下のいずれかを決める
  - 仕様に合わせて修正（例: 時間窓、フィード一覧、TOP_Nなど）
  - 現行実装を仕様として許容（例: RSSリーダー互換性のためのpubDate方針など、合理性がある場合）
  - 保留（判断材料不足。期限と追加調査タスクを付ける）
- 修正する差分は、影響範囲（テスト/ドキュメント/運用）をセットで更新する

達成度
- A: “合わせる”差分はすべて修正完了し、テストが更新されている
- B: “合わせる”差分のうち高優先のみ完了（残件は期限付きで保留）
- C: “合わせる/許容/保留”の分類だけ完了（合意形成まで到達）

### フェーズ3: メトリクス導入（最適化のための計測）
最適化は「測定→仮説→変更→再測定」を原則とし、先に“計測可能な状態”を作る。

- 収集するメトリクス（最低限）
  - 取得: フィード数、取得件数、パース件数、正規化後件数、時間窓通過件数、重複除去数（[src/fetcher.py](../src/fetcher.py)、[src/main.py](../src/main.py)）
  - スコア: 対象件数、APIコール回数（バッチ/個別別）、再試行回数、フォールバック回数、キャッシュヒット率（[src/scorer.py](../src/scorer.py)）
  - 出力: RSSアイテム数、出力サイズ、生成時間（[src/rss_builder.py](../src/rss_builder.py)）
  - 全体: 実行時間（総計、フェーズ別）、エラー種別別カウント（[src/logging_utils.py](../src/logging_utils.py)）
- 記録方法（最小実装の方針）
  - まずはログ出力で一貫したキー名を使う（後から機械集計しやすい形に寄せる）
  - 追加で必要なら、.cache配下へ実行単位のJSON Linesを保存する（既存のキャッシュ方針と干渉しない前提で検討。キャッシュ関連は [src/scorer.py](../src/scorer.py)、ディレクトリは [src/config.py](../src/config.py) の `CACHE_DIR`）

達成度
- A: 主要メトリクスが毎回出力され、最適化のBefore/After比較ができる
- B: 主要メトリクスが一部でも継続記録される（スコア系＋全体時間は必須）
- C: メトリクス定義のみ確定（計測は次フェーズ）

## テスト方針（最小で効く回帰防止）
- 既存テストの位置づけを維持しつつ、差分の“合意した挙動”を固定する
  - ランキング: [tests/test_ranking.py](../tests/test_ranking.py) と [src/ranking.py](../src/ranking.py)
  - RSS: [tests/test_rss_builder.py](../tests/test_rss_builder.py) と [src/rss_builder.py](../src/rss_builder.py)
  - フォールバック: [tests/test_scorer_fallback.py](../tests/test_scorer_fallback.py) と [src/scorer.py](../src/scorer.py)
- 追加が必要になりやすい領域（差分次第で最小追加）
  - 時間窓フィルタ: [src/main.py](../src/main.py) の `filter_recent_articles`
  - スコア合計ロジック: [src/models.py](../src/models.py) の `ScoreResult.total`
  - フィード一覧/設定既定値の整合: [src/config.py](../src/config.py)

## リスクと対策
- 仕様が複数文書に分散: 差分管理を唯一の判断ログにし、結論の出所を固定する（[docs/spec_diffs.md](spec_diffs.md)）。
- LLM出力の揺れ: 検証・再試行・フォールバックの分岐を整理し、テストは“構造”と“範囲”を中心に固定する（[src/scorer.py](../src/scorer.py)）。
- RSS互換性: pubDate/descriptionの扱いはRSSリーダー差があるため、現行挙動を変える場合は必ず差分に記録し、テストで固定する（[src/rss_builder.py](../src/rss_builder.py)）。

## 進捗管理（チェックリスト）
- 差分管理が作成・初期入力されている: [docs/spec_diffs.md](spec_diffs.md)
- “合わせる/許容/保留”が全差分で決まっている
- “合わせる”差分に対応するテスト更新が完了している
- 実行メトリクスが継続的に取得できる
